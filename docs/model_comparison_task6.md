# Задание 6 — Сравнение разных ИИ-моделей

**Проект:** Сервис аренды квартир (Вариант 26)
**Цель:** сравнить минимум 3 разные ИИ-модели на одной сложной задаче из предметной области.

---

## Задача

Реализовать **алгоритм ранжирования квартир по релевантности** для арендатора. Функция получает на вход:

- город и желаемые даты заселения/выезда;
- бюджет (максимальная цена за ночь);
- предпочтения по числу комнат и количеству гостей;
- историю прошлых бронирований арендатора (для выявления его предпочтений по районам/арендодателям).

На выход — отсортированный список квартир со скором релевантности (0–1) с учётом:

1. Цены (близость к бюджету, но не выше);
2. Среднего рейтинга по отзывам (только завершённые брони);
3. Загруженности квартиры за последние 90 дней (популярность);
4. Свежести последнего отзыва (затухание во времени);
5. Совпадения с историей арендатора (тот же город / тот же арендодатель — бонус).

**Промпт** (одинаковый для всех моделей):

> Ты — senior Python разработчик в сервисе аренды квартир (стек: FastAPI + SQLAlchemy 2.0). Реализуй функцию `rank_apartments(db, tenant_id, city, check_in, check_out, max_price, guests, rooms) -> list[ApartmentScore]`. Логика скоринга:
> - вес 0.30 — близость цены к бюджету (формула затухания);
> - вес 0.25 — средний рейтинг по отзывам только из бронирований со статусом completed;
> - вес 0.15 — нормированная загруженность за 90 дней (число completed-броней / число дней в окне);
> - вес 0.15 — свежесть отзывов (экспоненциальное затухание с полупериодом 60 дней);
> - вес 0.15 — персонализация по истории арендатора (тот же город +0.5, тот же арендодатель +0.5).
>
> Фильтры до скоринга: is_active=True, доступные даты (нет пересекающихся pending/confirmed броней), цена <= max_price, capacity >= guests, rooms == rooms (если указан).
>
> Требования к коду: type hints, обработка пустых случаев, без N+1 запросов, объяснение формулы для скора.

## Решения

### Claude Sonnet 4.6

Сгенерировал решение в одну итерацию. Структурно:

- Отдельный dataclass `ApartmentScore(apartment, score, breakdown)` — позволяет дебажить вклад каждого фактора.
- Фильтрация на уровне SQL через `notin_(busy_subquery)` для дат — без N+1.
- Использует `func.coalesce(func.avg(Review.rating), 0)` чтобы квартиры без отзывов не выпадали из выборки.
- Персонализация — отдельный шаг после первичной выборки: пересекается с `tenant.bookings.apartments`.
- Каждой формуле скоринга — однострочный комментарий «почему так» (например, для свежести отзывов — почему `2^(-age_days/60)`).

```python
# Фрагмент: близость цены к бюджету (вес 0.30)
# Линейное затухание: ровно бюджет -> 1.0, в 2 раза дешевле -> 0.5
# (дешевле бюджета не штрафуем сильно, бюджет — верхняя граница)
def _price_score(price: Decimal, budget: Decimal) -> float:
    if price > budget:
        return 0.0
    ratio = float(price / budget)
    return 0.5 + ratio / 2
```

**Корректность:** код запустился сразу, тесты на 5 синтетических квартирах дали ожидаемый порядок. OK
**Полнота:** все 5 факторов, фильтры до скоринга, обработка пустого случая. OK
**Безопасность:** параметризованные запросы, без сырого SQL. OK
**Читаемость:** разбито на 5 приватных функций (по одной на фактор), главный `rank_apartments` короткий. OK

### GPT-4 Turbo

Первая итерация работала, но содержала **N+1 проблему**: средний рейтинг и загруженность считались в Python в цикле по квартирам. На 100 квартирах — 200 дополнительных SQL-запросов.

После второго промпта («перенеси avg(rating) и count(bookings) в один SQL-запрос с GROUP BY») переписал на агрегаты через подзапрос, как у Claude.

Стиль кода — менее «питоничный»: все формулы скоринга внутри одной длинной функции без разбиения, без комментариев к весам.

```python
# Фрагмент: загруженность (вес 0.15) — после исправления
sub = (
    db.query(
        Booking.apartment_id,
        func.count(Booking.id).label('cnt'),
    )
    .filter(Booking.status == BookingStatus.completed,
            Booking.check_out >= since)
    .group_by(Booking.apartment_id).subquery()
)
```

**Корректность:** работает после второй итерации. ЧАСТИЧНО
**Полнота:** все 5 факторов, но фильтр `rooms == rooms` пропустил (нужно было напомнить). ЧАСТИЧНО
**Безопасность:** параметризованные запросы. OK
**Читаемость:** одна функция на 70+ строк, магическое число `60` без объяснения. ПЛОХО

### Qwen3.5:27b (локальная)

Заняло **3 итерации**.

1. Первая попытка — использовал `Apartment.bookings` как relationship, но из-за `lazy='select'` по умолчанию это давало N+1.
2. После уточнения «без N+1» — переписал на raw SQL через `text()`. Работало, но обходило ORM, что неприемлемо в проекте.
3. После третьего уточнения «используй SQLAlchemy ORM, не raw SQL» — выдал корректный вариант через `subquery()`, похожий на Claude, но без dataclass для breakdown (только `tuple[Apartment, float]`).

Свежесть отзывов забыл — пришлось напомнить четвёртой репликой.

Время на каждую итерацию: **~120–180 сек** (заметно медленнее облачных).

**Корректность:** только после 3 итераций. ЧАСТИЧНО
**Полнота:** забыл свежесть, нужен был дополнительный промпт. ПЛОХО
**Безопасность:** в первой версии — конкатенация строк в `text()` (потенциальная SQL-инъекция), исправлено только после явного указания. ПЛОХО
**Читаемость:** код компактный, но без type hints на возвращаемых значениях. ЧАСТИЧНО

## Выводы

1. **Claude Sonnet 4.6** дал лучший результат с первой попытки: чистая структура, безопасный SQL, объяснения к скорингу. Это типичная картина для задач, где важна архитектурная аккуратность (разбиение на функции, dataclass для дебага).

2. **GPT-4 Turbo** уверенно работает, но требует явных указаний по производительности (без N+1) и не комментирует свои решения. Подходит для генерации «рабочего кода» без претензий на чистоту.

3. **Qwen3.5:27b (локальная)** справилась, но с большим числом итераций и опасной первой версией (SQL-инъекция через `text()`). Без code review её код в продакшен брать нельзя. Скорость — критическое ограничение: 3 итерации × 150 сек = ~8 минут против ~10 секунд у Claude.

4. **Главный риск локальной модели** — она охотнее обходит ORM в пользу raw SQL и упускает требования безопасности. На задачах с базой данных это особенно опасно.

5. **Где локальная модель оправдана**: офлайн-разработка, NDA-проекты с запретом на отправку кода в облако, бюджетные ограничения. Для остальных случаев облачные модели выигрывают по всем критериям, кроме приватности.

**Итоговая рекомендация для проекта аренды квартир:** Claude Sonnet 4.6 для архитектурных задач и сложной бизнес-логики (скоринг, расчёт цен, поиск конфликтов броней); GPT-4 Turbo как второе мнение или для генерации тестов; локальная Qwen — только для черновиков и неприоритетных задач.
